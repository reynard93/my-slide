Implementation notes

This file records how the slide implementation interprets the brief while I keep iterating.

Design decisions

  • Deck register: treated as a technical sharing deck, not a product landing page. Visual direction: dark workshop / terminal lab with warm text, cyan/lime/amber accents, and diagrammatic layouts.
  • Narrative weight: codebase context is the main arc. NotebookLM and web debugging are included as optional case studies, not the center of gravity.
  • Starter repo location: kept inside slides/context-optimization/assets/context-optimization-starter/ and bundled as context-optimization-starter.zip.
  • Notes location: this file lives in slide assets to avoid adding extra top-level project docs.
  • Visual polish: replaced repeated tiny section eyebrows with a small route chip beside titles. This keeps orientation without the standard AI landing-page scaffold.

Isolation decisions

  • No setup mutation: I am not running install commands, global hooks, graphify install, Headroom proxies, AgentMemory servers, Browserbase sessions, or NotebookLM login.
  • Starter defaults: the sync script now defaults to .starter-output/drive-context/ inside the starter folder. It no longer writes to ../drive-context unless a user explicitly passes that path.
  • Temp verification: starter verification extracts the zip into a temp directory and runs the demo there, so the take-home flow is tested without touching your working setup.

Fact-check sources used

  • safishamsi/graphify: README confirms queryable graph over code/docs/PDF/images/video and outputs graph.html, GRAPH_REPORT.md, graph.json.
  • parcadei/tldr-code: README confirms structure, calls, impact, dead, health, security, quality commands, JSON output, and 18 languages.
  • rjkaes/trueline-mcp: README confirms AST outline, targeted reads, hash-verified edits, atomic batches, and stale edit rejection.
  • rtk-ai/rtk and JuliusBrussee/caveman: RTK filters command output; Caveman targets output-token reduction.
  • chopratejas/headroom and Headroom limitations: README frames it as local proxy/wrapper/SDK/MCP compression for tool outputs, logs, files, RAG chunks, and conversation history; limitations note little value for short chats and code-only sessions.
  • rohitg00/agentmemory: README frames it as persistent memory for coding agents via MCP/hooks/REST across many clients.
  • browserbase/stagehand and Browserbase Stagehand page: confirms act/extract/observe/agent and optional Browserbase cloud features.
  • teng-lin/notebooklm-py: README explicitly says unofficial and uses undocumented Google APIs.
  • Helping agents debug webapps: supports the console-log bridge case study.

Deviations

  • Impeccable helper script: the local helper path .claude/skills/impeccable/scripts/context.mjs is not present in this repo. I used the loaded skill references directly instead of running local helper setup.
  • Stagehand example: included as a skeleton only. It is not executed during verification because that would require dependencies and credentials outside the isolated starter.
  • TDD: there is no application behavior change in the slide runtime. For the starter scripts, I used behavior checks in a temp extraction rather than adding a test framework dependency.

Tradeoffs

  • Plain scripts over dependencies: the starter uses Node and Bash so people can inspect it quickly. The tradeoff is fewer platform niceties.
  • Fact claims: slide copy avoids brittle benchmark numbers unless they are clearly repo claims. This keeps the talk useful if numbers change later.
  • Slide density: tool facts are compressed into a receipts slide. That gives credibility without turning the deck into a README tour.
  • Headroom placement: included as an optional live compression layer, not a replacement for Graphify's map or AgentMemory's recall. It earns its slot only when input tokens dominate.
